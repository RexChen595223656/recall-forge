# v4.0 测试方案总纲

## 一、测试策略

### 测试金字塔

```
        ┌─────────┐
        │ 手动探索  │  ← 验收清单逐条走查
       ┌┴─────────┴┐
       │  E2E 流程  │  ← Playwright 浏览器自动化
      ┌┴───────────┴┐
      │ API 合约测试  │  ← pytest + httpx 接口测试
     ┌┴─────────────┴┐
     │  后端单元测试   │  ← pytest 纯函数/模块测试
    └───────────────┘
```

### 各层职责

| 层级 | 工具 | 覆盖目标 | 运行时间 | 失败标准 |
|------|------|---------|---------|---------|
| 单元测试 | pytest | 后端核心逻辑 100% | < 10s | 任一条失败 |
| API 合约 | pytest + httpx | 所有端点正常+异常+边界 | < 30s | 任一条失败 |
| E2E | Playwright | 关键用户旅程 100% | < 5min | 核心流程中断 |
| 手动探索 | Chrome 无痕 | 验收清单 84 条全覆盖 | ~30min | 任一 P0 项失败 |

### 不测什么

- AI 出题质量（人工评估，不在此测试体系）
- ChromaDB 向量检索精度（第三方库，信任其正确性）
- LLM 响应格式（已在 Prompt 层约束，回归时人工抽查）
- 浏览器兼容性（仅测 Chrome，多浏览器不在范围）

## 二、测试环境

| 项目 | 要求 |
|------|------|
| 操作系统 | macOS 15+ |
| 浏览器 | Chrome 最新稳定版（headless） |
| Python | 3.14 |
| Node.js | 22+ |
| 数据库 | SQLite（文件模式，每次测试前删除 quiz.db） |
| 向量库 | ChromaDB（持久化模式，每次测试前删除 data/chroma/） |
| 网络 | 需能访问 DeepSeek API（`api.deepseek.com`），超时 60s |
| 并发 | 单用户，不测并发 |

### 环境初始化

```bash
# 每次测试运行前执行
rm -f /path/to/recall-forge/backend/data/quiz.db
rm -rf /path/to/recall-forge/backend/data/chroma/

cd /path/to/recall-forge/backend
python3 -m uvicorn main:app --port 8000 &

cd /path/to/recall-forge/frontend
npm run dev &

# 等待就绪
until curl -s http://localhost:8000/api/health; do sleep 1; done
until curl -s -o /dev/null http://localhost:3000; do sleep 1; done
```

## 四、测试数据

### 标准测试材料

| 材料 | 文件名 | 内容特征 | 预期行为 |
|------|--------|---------|---------|
| 极短 | `too-short.txt` | 49 字 | 上传被拒 |
| 短文 | `short.txt` | 50-200 字，1 chunk | 生成 3 题 |
| 中文 | `medium.txt` | 300-800 字，3-4 chunks | 生成 9-12 题 |
| 长文 | `long.txt` | 3000+ 字，10+ chunks | 生成 30 题（上限） |
| PDF | `sample.pdf` | 任意 PDF | 解析后正常出题 |

### 测试材料内容

**too-short.txt** (49 字):
```
AI产品经理是人工智能时代的新角色。他们需要理解技术边界，设计产品方案。
```

**short.txt** (准备在测试脚本中动态生成，确保 ≥50 字且 ≤200 字):
```
AI产品经理的核心能力包括理解大模型能力边界、设计RAG架构、工程化Prompt系统、建立多层评估体系和构建数据飞轮。这些能力与传统PM的需求分析、用户体验设计能力互补，形成AI时代产品经理的完整能力模型。掌握这些能力的PM能更好地与技术团队协作，设计出真正有价值的AI产品。
```

**medium.txt** (准备在测试脚本中动态生成，确保 300-800 字):
包含 4-5 个独立段落，每个段落 ≥100 字的 AI 相关知识内容。

**long.txt** (准备在测试脚本中动态生成，确保 3000+ 字):
包含 10+ 个独立段落的完整技术文档。

## 六、测试执行流程

```
┌─────────────┐
│ 1. 环境准备   │  rm -f quiz.db && rm -rf chroma/
└──────┬──────┘
       ↓
┌─────────────┐
│ 2. 单元测试   │  pytest tests/unit/ -v
│    < 10 秒   │  全部通过 → 继续
└──────┬──────┘
       ↓
┌─────────────┐
│ 3. API 测试  │  pytest tests/api/ -v
│    < 30 秒   │  全部通过 → 继续
└──────┬──────┘
       ↓
┌─────────────┐
│ 4. E2E 测试  │  pytest tests/e2e/ -v --timeout=120
│    < 5 分钟  │  核心用例全部通过 → 继续
└──────┬──────┘
       ↓
┌─────────────┐
│ 5. 手动验收   │  Chrome 无痕窗口
│    ~30 分钟  │  逐条走查 F-13 验收清单
└──────┬──────┘
       ↓
┌─────────────┐
│ 6. 测试报告   │  汇总通过/失败/阻塞项
└─────────────┘
```

### 各阶段失败处理

| 阶段 | 失败 | 处理 |
|------|------|------|
| 单元 | 任一条 | 修复后从第 1 步重新开始 |
| API | 任一条 | 修复后从第 2 步重新开始 |
| E2E | 核心用例 | 修复后从第 3 步重新开始 |
| 手动 | P0 项 | 修复后从第 4 步重新开始 |

## 七、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| DeepSeek API 不可用 | 出题相关测试失败 | Mock LLM 响应做离线测试（后续迭代） |
| Playwright Chromium 版本不匹配 | E2E 无法运行 | 固定 playwright 版本，CI 环境用 Docker |
| 前端端口被占用 | 服务启动失败 | 测试脚本自动 kill 占用进程 |
| ChromaDB 状态残留 | 测试数据污染 | 每次测试前删除 chroma/ 目录 |
