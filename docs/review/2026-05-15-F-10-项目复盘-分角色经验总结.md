# RecallForge（知锻）项目复盘 · 分角色经验总结

## 项目概要

从生成式学习工具的概念出发，做了 RAG + SM-2 间隔重复 + 批量出题 + 多格式输入 + 诊断式出题的完整版。全栈自建（Next.js 16 + Python FastAPI + ChromaDB + DeepSeek），历时 2 天。

---

## 一、AI 产品经理经验

### 1.1 竞品分析要挖"没做什么"
- **踩坑**：刚开始想完全复刻，但原版有三个明显产品缺陷：逐题等待（体验差）、流式暴露答案（作弊风险）、出题无去重（内容重复）
- **经验**：竞品分析不是"他做了什么"，而是"他没做什么"。三个缺陷就是三个差异化切入点
- **原则**：**分析竞品 → 找到缺失 → 自己的差异化方案**，不要照搬

### 1.2 设置默认值就是产品策略
- **踩坑**：出题设置 5 个维度都有默认值，但没有精心设计默认值的"黄金路径"
- **经验**：默认值决定了 90% 用户的第一体验。默认值应该是：
  - 提取方式：原文提取（最安全，新手友好）
  - 难度：中等（不打击也不无聊）
  - 题型：单选（认知负担最低）
  - 数量：5 题（快速闭环，不吓人）
  - 反馈：即时（即时满足感）
- **原则**：**默认值 = 产品策略，不是随便填的**

### 1.3 空态设计决定激活率
- **踩坑**：最初只有一行灰色文字，后来才加入示例材料一键导入
- **经验**：空态是用户的第一印象。必须有：清晰的下一步指引 + 一键体验路径（示例数据）
- **原则**：**空态 = 转化的关键时刻，不能只有一句话**

### 1.4 复习流程的透明度
- **踩坑**：SM-2 算法对用户是黑盒——不知道为什么只有 1 题、不知道为什么明天才出现
- **经验**：AI 产品的算法逻辑必须对用户可见。加了"查看全部"开关和到期时间展示后，用户才理解发生了什么
- **原则**：**算法驱动的功能，必须让用户看到"为什么"**

### 1.5 产品名要有差异化
- **踩坑**：最初的名字太窄，后改为"RecallForge（知锻）"
- **经验**：名字决定了用户的第一认知。"刷题"是把产品限定在了考试场景，"知识内化引擎"打开了更大的使用场景
- **原则**：**产品名 = 定位声明，不要用功能描述当名字**

## 二、开发经验

### 2.1 DeepSeek Anthropic 兼容 ≠ 完全相同
- **问题**：DeepSeek 流式返回 `thinking_delta`（思考） + `text_delta`（输出）双 content block，而标准 Anthropic 只有 `text_delta`
- **表现**：SSE 流式返回了大量空字符串，题目永远不出现
- **修复**：区分三种 delta 类型，只向前端推送 `text_delta`
- **原则**：**接入新 AI provider 时，先用 curl 跑一遍原始流式输出，确认格式再写解析逻辑**

### 2.2 Python .format() + JSON 模板 = 灾难
- **问题**：QUIZ_PROMPT 中 JSON 示例的花括号 `{"question": "..."}` 被 `.format()` 解析为占位符
- **表现**：KeyError，服务崩溃
- **修复**：JSON 部分的花括号双写 `{{` `}}`
- **原则**：**包含 JSON 示例的 Prompt 模板用 `.replace()` 或全文双花括号转义，不要用 `.format()`**

### 2.3 Tailwind v4 配置方式完全改变
- **问题**：Tailwind v4 使用 CSS `@theme inline` 替代 `tailwind.config.ts`
- **表现**：原计划写的 `tailwind.config.ts` 不起作用
- **修复**：在 `globals.css` 中定义自定义 token
- **原则**：**新版本框架的 breaking changes 要第一时间查文档，不要假设和旧版一样**

### 2.4 React setState 在 render 阶段 = 无限循环
- **问题**：`setSessionStats()` 直接在 `if (phase === "review")` 分支里调用
- **表现**：页面持续闪烁，无限重渲染
- **修复**：改为 `useRef` 存储 session 数据
- **原则**：**任何 setState 必须只在事件处理或 useEffect 中调用，绝不能在 render 路径直接调用**

### 2.5 startsWith 方向错误
- **问题**：`userAnswer.startsWith(q.answer)` — 用户答案"B"去 match 正确选项"B. Agent架构"，方向反了
- **表现**：用户答对了显示错误
- **修复**：改为 `q.answer.startsWith(userAnswer)`，同时后端保存时标准化答案只取首字母
- **原则**：**字符串比较类的函数，写完后立刻用正反两个用例验证**

### 2.6 级联数据删除
- **问题**：删除材料时只删了材料和 ChromaDB chunks，没删关联的题目、答题记录、复习卡片
- **表现**：知识库删了，复习里还有残留题目
- **修复**：delete 端点手动级联删除：ReviewCards → Attempts → Questions → Material
- **原则**：**删除操作必须考虑级联影响，ORM 的 CASCADE 配置比手动删除更可靠**

### 2.7 Python 3.14 兼容性
- **问题**：Python 3.14 是极新版，pydantic-core 等包无预编译 wheel
- **表现**：pip install 构建失败
- **修复**：系统已预装部分包，跳过版本锁定使用兼容版本
- **原则**：**新项目锁定 Python 3.11/3.12，不追最新版**

### 2.8 Turbopack 缓存损坏
- **问题**：`.next` 目录缓存损坏导致 Turbopack 反复崩溃
- **表现**：页面持续闪烁，看起来像在不停刷新
- **修复**：删除 `.next` 重新编译
- **原则**：**遇到莫名其妙的构建/运行时问题，先清缓存再排查**

## 三、测试经验

### 3.1 不要攒到最后才测
- **踩坑**：后端写完所有 API 才测试，前端写完所有页面才联调
- **经验**：写完一个 API 端点 → 立刻 curl 测试 → 确认数据正确 → 再写下一个
- **原则**：**每完成一个功能单元（API端点/页面/组件），立即验证，不攒到最后**

### 3.2 SSE 流式测试需要特殊方式
- **踩坑**：普通 curl 请求看不到 SSE 流式输出
- **经验**：`curl -N --max-time 30` 可以看到 SSE 事件流
- **原则**：**流式 API 的测试方式和普通 REST API 不同，需要专门的测试脚本**

### 3.3 headless 浏览器点击可能失效
- **踩坑**：agent-browser 的 `@eN` ref 点击在嵌套 span 的 button 上失效
- **经验**：改用 JS eval + `querySelector` + `.click()` 更可靠
- **原则**：**自动化测试工具的 ref 点击不是 100% 可靠，复杂 DOM 结构用 JS 直调**

## 四、设计经验

### 4.1 按钮 hover 要全项目统一检查
- **踩坑**：多处按钮没有 hover 效果，用户反馈后才逐个补
- **经验**：定义全局按钮 variant（primary/secondary/ghost/danger），统一 hover 行为
- **原则**：**设计 token 体系要在项目初期定义好，包括交互态（hover/focus/active/disabled）**

### 4.2 文件上传必须有选中和进度反馈
- **踩坑**：最初的 `<input type="file">` 是个裸控件，选没选中完全不知道
- **经验**：拖拽区域 + 文件名/大小展示 + 上传进度条，三者缺一不可
- **原则**：**任何用户操作都必须有"确认收到"的视觉反馈**

### 4.3 侧边栏 vs 顶栏要根据页面数决定
- **踩坑**：3 个页面用了侧边栏，有点过度设计
- **经验**：少于 5 个核心页面的应用，顶栏导航更简洁；多于 5 个或需要二级导航才用侧边栏
- **原则**：**导航复杂度要和信息架构复杂度匹配，不要为了好看而过度设计**

## 五、项目监管经验

### 5.1 执行命令前检查当前目录
- **踩坑**：至少 5 次在错误目录执行 `npm run build` 或 `python3 -m uvicorn`
- **经验**：用绝对路径或先 `pwd` 确认
- **原则**：**每条命令前确认 cwd，长任务用 `cd /abs/path && cmd`**

### 5.2 改完代码要重启服务
- **踩坑**：改了后端代码，Turbopack 不会重启 Python 进程，导致前端拿到旧 API 响应
- **经验**：Python 后端不会热更新，每次改完要手动 `pkill + 重启`
- **原则**：**改后端 → 重启后端 → 验证 API → 前端联调，不要跳过重启**

### 5.3 项目改名要全局搜索替换
- **踩坑**：目录名改了但代码里的引用没改全
- **经验**：`grep -r` 全局搜索旧名 → `sed` 批量替换 → 重新构建验证
- **原则**：**改名不仅仅是 `mv`，是代码 + 文档 + 记忆全面替换**

### 5.4 全局记忆要及时更新
- **踩坑**：踩了坑没有立刻记下来，下次还会犯
- **经验**：每次踩坑立刻写进项目 `docs/review/` 和全局 `memory/`
- **原则**：**踩坑 → 修复 → 记录 → 下次不踩，形成闭环**
