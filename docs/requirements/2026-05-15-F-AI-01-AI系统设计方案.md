# RecallForge AI 系统设计方案

## 1. 模型选型

| 维度 | 决策 | 理由 |
|------|------|------|
| 主模型 | DeepSeek V4 Pro | 中文能力强，成本低，已有 Anthropic 兼容端点 |
| thinking 模式 | **禁用** | 出题任务不需要深度推理，thinking 增加 9s 延迟且产出无价值内容 |
| max_tokens | 批量 2048（5题）/ 单题 1024 | 5 题 JSON 约 1500-2000 tokens |
| 温度 | 默认（不设） | JSON 输出需要确定性，低温度更稳定 |
| 备选模型 | Claude Opus 4 | 如果 DeepSeek 不稳定，切换成本低（同一端点格式） |

## 2. Prompt 架构

### 2.1 系统架构

```
[材料 chunk] → Prompt 模板 → LLM → JSON 数组 → [题目保存]
                  ↑
           [五维配置参数]
```

### 2.2 Prompt 模板（当前版本）

```
你是一个专业的出题老师。基于以下学习材料，生成 {count} 道题目。

模式：{mode_instruction}
难度：{difficulty_instruction}
题型：{type_instruction}

额外要求：
- 每题提供解析，解释为什么正确答案是对的、错误答案为什么不对
- 给每道题打2-3个标签（知识点、难度等）
- 题目之间不要重复，覆盖材料的不同知识点

输出严格的JSON数组（不要markdown代码块标记）。

学习材料：{chunks_text}

请生成 {count} 道题目：
```

### 2.3 模板参数化规则

| 参数 | 来源 | 可选值 | 如何影响 Prompt |
|------|------|--------|----------------|
| count | 用户设置 | 5/10/15/20 | 替换 `{count}` |
| mode | extract/expand | 原文提取/知识拓展 | 切换模式指令段落 |
| difficulty | easy/medium/hard | 简单/中等/困难 | 切换难度指令段落 |
| question_type | single/multi/single,multi | 单选/多选/混合 | 切换题型指令段落 |
| chunks_text | RAG 检索 | 动态 | 拼接 Top-K chunk 内容(每 chunk 截断 500 字) |

### 2.4 输出格式约束

**单选：**
```json
{
  "question": "题干",
  "options": ["A. 选项一", "B. 选项二", "C. 选项三", "D. 选项四"],
  "answer": "B",
  "explanation": "解析内容",
  "tags": ["标签1", "标签2"]
}
```

**多选：**
```json
{
  "question": "题干",
  "options": ["A. 选项一", "B. 选项二", "C. 选项三", "D. 选项四", "E. 选项五"],
  "answers": ["A", "C"],
  "explanation": "解析内容",
  "tags": ["标签1", "标签2"]
}
```

**混合模式：** JSON 数组里混合包含上述两种格式，客户端根据 `answers` 字段是否存在判断多选。

## 3. RAG 策略参数

| 参数 | 值 | 理由 |
|------|-----|------|
| Chunk 大小 | 300 字 | 匹配出题需要的信息粒度 |
| Overlap | 50 字 | 避免关键信息被截断 |
| 最小 Chunk | 20 字 | 过滤无意义短段落 |
| 检索方式 | 随机（改版前）/ 优先未使用（改版后） | 避免重复出题 |
| 检索数量 | count×3（最少10） | 给 LLM 足够的素材选择 |
| Chunk 截断 | 每 chunk 截取前 500 字送 LLM | 控制 Prompt 长度 |
| 向量库 | ChromaDB 内置 embedding | 零配置，离线可用 |
| 去重策略 | exclude_ids 参数 | 已出过题的 chunk 不再选 |

## 4. Streaming 策略

### 4.1 为什么不用流式展示

- **安全问题：** JSON 原文流式展示会提前暴露 `"answer": "B"`，等于作弊
- **体验问题：** JSON 格式对用户无意义，展示乱码不如动画

### 4.2 当前策略：内部流式 + 前端动画

```
后端: 流式获取 LLM 输出 → 累积 full_text → message_stop 时解析 JSON → 一次性返回
前端: 显示生成动画(圆点+文字轮播) → 收 complete 事件 → 渲染题目
```

**心跳机制：** 每 3 秒发一次 `progress` 事件，保持 SSE 连接活跃，防止超时断开。

### 4.3 前端 SSE 解析规则

```
event type:
  - content_block_start(type="text") → 标记进入正文区块
  - content_block_delta(type="text_delta") → 累积文本，不发前端
  - content_block_delta(type="thinking_delta") → 忽略(DeepSeek特有)
  - content_block_delta(type="signature_delta") → 忽略
  - message_stop → 解析累积文本为JSON → 发送 complete 事件
  - progress(心跳) → 前端保持连接
```

## 5. 答案处理策略

### 5.1 答案标准化流程

```
LLM 输出 answer 字段
    ↓
可能格式: "B" | "B." | "B. Agent架构" | "b"
    ↓
后端保存时标准化: answer[0].upper() → "B"
    ↓
前端提交时: option.charAt(0).toUpperCase() → "B"
    ↓
判定: 正确答案.startsWith(用户选择)
```

### 5.2 错误处理

| 场景 | 处理 |
|------|------|
| LLM 返回的 answer 是空字符串 | 保存原值，判题时走本地比对兜底 |
| LLM 返回的 options 不足 4 个 | 不拦截，LLM 的约束在 Prompt 里 |
| options 不以字母开头 | 正常处理，`charAt(0)` 取第一个字符 |
| 多选的 answers 是单字母字符串 | 兼容：当 `answers.length === 1` 时视同单选 |

## 6. Fallback 链

```
第1层: DeepSeek V4 (thinking disabled) — 主模型
    ↓ 失败(HTTP错误/超时30s)
第2层: 重试一次 DeepSeek V4
    ↓ 失败
第3层: 返回错误提示 + 建议用户稍后重试
```

**当前无自动切换备选模型**，因为 DeepSeek 稳定性在可接受范围内，切换逻辑增加复杂度不划算。

## 7. 成本控制

| 机制 | 说明 |
|------|------|
| 批量出题 | 5 题一次 API 调用，减少请求次数 |
| thinking disabled | 减少 output token（thinking token 不计入最终输出但会计费） |
| Chunk 去重 | 避免对同一 chunk 重复出题浪费 API 调用 |
| 每日 200 题软限制 | config.py 中 DAILY_QUESTION_LIMIT，前端可提示 |
