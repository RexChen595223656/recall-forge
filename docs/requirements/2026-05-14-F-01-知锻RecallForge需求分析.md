# v1.0 知锻(RecallForge)增强版 需求分析文档 (PRD)

## 1. 需求背景
- **需求来源：** 产品规划（复刻学习 + 功能增强）
- **背景描述：** 信息工作者每天被海量文档淹没，阅读后难以转化为可验证的知识资产。传统"看完即忘"的学习模式缺乏闭环反馈。AI 牛马刷题通过"材料→出题→答题→复习"闭环，将被动阅读转化为主动学习。
- **关联项目：**  知锻(RecallForge)（无）

## 2. 版本目标
- **业务目标：** 打造一个完整的学习闭环工具，验证"AI 生成式学习 + 间隔重复"的产品假设
- **核心指标：** 日活用户、单次学习时长、题目生成成功率、复习留存率
- **约束条件：** 单人开发，AI 辅助（Vibe Coding），2周内完成 MVP

## 3. 需求分析（双轨模型）

### 3.1 传统分析轨道

| 用户角色 | 场景 | 现状 | 痛点 | 定位 | 价值 | 功能支撑 | 数据支撑 |
|---------|------|------|------|------|------|---------|---------|
| 学习者 | 阅读长文档后想自测理解程度 | 手动做笔记、找题刷 | 材料与题目脱节，无针对性 | AI 从材料中自动出题 | 学习效率提升，知识内化 | 材料上传 + 题目生成 | 出题准确率、用户答题正确率 |
| 复习者 | 考前/面试前系统复习 | 随机翻看笔记 | 缺乏科学排期，遗忘率高 | 艾宾浩斯间隔重复推送 | 记忆力提升，复习效率提升 | 复习中心 + SM-2 排期 | 复习完成率、记忆保持率 |
| 知识管理者 | 想积累个人知识资产 | 文档散落各处 | 知识不成体系，无法量化 | RAG 知识库 + 掌握度仪表盘 | 知识资产可视化，学习有据可查 | 知识库管理 + 统计仪表盘 | 知识库增长、掌握度趋势 |

### 3.2 AI 可行性分析轨道

- **AI 解决还是规则解决：** AI 解决（题目生成、答案判定语义相似度）；规则解决（SM-2排期、去重、格式化）
- **数据基础：** 用户上传的文档即数据源，无需预训练数据
- **模型方案：** RAG + LLM（检索增强生成题目）
- **评估标准：** 题目相关性 > 80%、答案唯一性 > 90%、无幻觉捏造
- **冷启动策略：** 内置示例材料 + 示例题目，用户可立即体验
- **迭代路径：** 单选题 → 多选题/填空题 → 主观题 AI 评分 → 自适应难度


### 4.3 AI 功能验收标准

**【功能点】AI 动态出题**
- 输入条件：用户选择知识库中的材料，点击"开始刷题"
- 预期输出：4 选项单选题，含题干、选项、正确答案、解析
- 准确性标准：题目必须基于上传材料内容，无凭空捏造
- 失败处理：chunk 不足时提示"材料内容不足，请上传更多材料"
- 通过条件：连续生成 10 题，人工抽检 80% 以上与材料相关

**【功能点】答案判定**
- 输入条件：用户选择选项并提交
- 预期输出：正确/错误 + 解析
- 准确性标准：与预设答案对比，100% 准确
- 失败处理：记录异常，降级为手动对照
- 通过条件：判定逻辑测试通过

## 5. 数据需求
- 采集需求：用户上传材料内容、答题记录、复习行为
- 埋点需求：页面访问、生成耗时、答题耗时、复习完成率
- 评估数据需求：人工标注 50 题作为 ground truth 测试集

## 6. 开放问题与风险
- [风险] DeepSeek API 可能限流 → 加入请求队列 + 指数退避重试
- [风险] PDF 解析质量不稳定 → PyPDF2 降级为纯文本提取
- [风险] ChromaDB 嵌入式模式单机瓶颈 → MVP 阶段可接受，后续可迁至 server 模式
