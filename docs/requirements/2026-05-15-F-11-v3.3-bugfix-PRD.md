# v3.3 Bugfix PRD — 锻造台交互逻辑修复

> 版本：v3.3 | 日期：2026-05-15 | 类型：Bugfix 迭代

## 背景

v3.2 新增 10 项功能，但跳过了 PM 流程直接开发，导致每个功能都有交互逻辑 bug。本轮不新增功能，只修 bug。

---

## Bug 1：动态题数选择器与警告矛盾

**现状**：`total_chunks=1` 时，`Math.max(5, 2)=5`，只剩 `[5]` 可选，但警告仍提示"建议不超过 2 题"。

**根因**：最低题数硬编码为 5，与动态上限逻辑不匹配。

**修复方案**：
- 取消 `Math.max(5, ...)` 强制下限，改为 `Math.min(20, total_chunks * 2)`
- 数量按钮动态生成：从 1 到 maxCount，步长自适应（<=5 时每 1 一个，<=10 时每 2 一个，>10 时每 5 一个）
- 1 chunk → 可选 [1, 2]，2 chunk → 可选 [1, 2, 3, 4]，3 chunk → [1, 3, 5, 6] 以此类推
- 警告逻辑调整为 `config.count > total_chunks * 2`（上限外的不用警告，因为已经选不了）

**验收标准**：
- [ ] 1 chunk 时，数量选项不包含 5
- [ ] 3 chunk 时，可选 1/3/5/6
- [ ] 选择不超过上限时不显示警告
- [ ] 默认值自动适配：chunk 少于 5 时默认选 maxCount

## Bug 2：重新挑战模式无闭环

**现状**：点选项后只创建 Attempt 记录，不调 `recordReview`，SM-2 状态完全不更新。

**根因**：漏掉了 SM-2 评分调用。

**修复方案**：
- 答对：调 `recordReview(question_id, 3)`（相当于"很简单"），推进间隔
- 答错：调 `recordReview(question_id, 0)`（相当于"忘记了"），重置间隔
- 答对且 repetitions 达到一定次数（如 3 次以上）：自动从复习队列移除该题（可选，后续迭代）
- 重新挑战完成后显示本轮统计："重新挑战 N 题，正确 M 题"

**验收标准**：
- [ ] 答对后 ReviewCard 的 repetitions +1，next_review 延后
- [ ] 答错后 ReviewCard 的 repetitions 重置为 0，next_review 回到明天
- [ ] 重新挑战结束显示本轮统计
- [ ] 重新挑战模式中不提前显示正确答案

## Bug 3：已覆盖知识点始终为 0

**现状**：`saveQuestion` 不传 `chunk_id`，所有题目的 `chunk_id` 为空字符串。

**根因**：前端未保存 `chunk_id`，后端出题流程也未返回 `chunk_id`。

**修复方案**：
- 后端 `generate_quiz_stream` 返回题目时，附带题目所基于的 `chunk_id`（从 `get_random_chunks` 的返回中匹配）
- 前端 `saveQuestion` 时传入 `chunk_id`
- 简单匹配策略：用题目文本与 chunk 文本的相似度判定归属（或用 prompt 要求 LLM 返回 chunk 编号）

**简化方案**（推荐）：
- 出题时不精确匹配 chunk，改为在后端 save 时根据题目内容检索最相似的 chunk 并关联
- 或者更简单：每道题随机关联一个本次使用的 chunk_id（因为题目就是从这些 chunk 出的）

**验收标准**：
- [ ] 锻造 5 题后，`covered_chunks` 从 0 变为 >0
- [ ] 多次锻造后，`covered_chunks` 逐步增长
- [ ] "已覆盖 X/Y 个知识点" 数字不再永远是 0

## Bug 4 & 5：（合并修复）轮次聚类方向错误 + 时间标签颠倒

**现状**：attempts 按 DESC 排序，gap 检查用 `start_time - a.created_at`，导致簇内 `start_time` 是**最新**记录的时间，且连续 session 被碎片化。

**根因**：聚类方向与数据排序方向冲突。

**修复方案**：
- 改为按 ASC（从旧到新）排序
- 聚类时 `start_time` 取簇内第一条（最早）记录的时间
- 标签显示 `start_time`（即开始锻造的时间）
- 最后再 reverse 列表，保持"最新轮次排前面"的展示顺序

**验收标准**：
- [ ] 一轮从 15:20 到 15:35 的锻造，标签显示 "15:20"
- [ ] 相邻时间（<5 分钟间隔）的题目在同一轮
- [ ] >5 分钟间隔才算新轮次
- [ ] 最新轮次排在最上面

## Bug 6：SM-2 复习次数语义修正

**现状**：score < 2 时 `repetitions` 重置为 0。用户反复错一道题，永远显示"已复习 0 次"。

**根因**：SM-2 算法的 `repetitions` 含义是"连续成功次数"，但 UI 展示为"已复习次数"，语义不一致。

**修复方案**：
- 后端 ReviewCard 新增字段 `total_reviews`（总复习次数，只增不减）
- `recordReview` 时 `total_reviews += 1`，`repetitions` 保持原 SM-2 逻辑不变
- 前端展示改为"已复习 {total_reviews} 次 · 连续正确 {repetitions} 次"

**验收标准**：
- [ ] 第一次复习 score=0 → 显示"已复习 1 次 · 连续正确 0 次"
- [ ] 第二次复习 score=3 → 显示"已复习 2 次 · 连续正确 1 次"
- [ ] 第三次复习 score=0 → 显示"已复习 3 次 · 连续正确 0 次"

## Bug 7：测验模式缺少反馈文案和再来一轮

**现状**：quizReview 页没有"已加入复习"提示，也没有"再来一轮"按钮。

**修复方案**：
- quizReview 的错误题目卡片加"→ 已加入复习"文案（与 result 页一致）
- 加"再来一轮"按钮（与 result 页一致）

**验收标准**：
- [ ] 测验模式结果页错误题目标注"→ 已加入复习"
- [ ] 测验模式结果页有"再来一轮"和"返回"两个按钮

## Bug 8：上传分块反馈边缘情况

**现状**：chunk_count=0 时静默关闭，且 Material 类型不含 chunk_count。

**修复方案**：
- Material 接口加 `chunk_count?: number`
- chunk_count=0 时也显示提示："内容已上传，但内容较短未拆分知识点块"
- 修复 `uploading` 状态在成功后未重置的问题

**验收标准**：
- [ ] 上传短文（<50 字被拒绝前）成功时显示分块数
- [ ] chunk_count=0 时显示对应提示而非静默关闭
